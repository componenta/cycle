# Componenta Cycle

Интеграция Componenta Framework с Cycle ORM/DBAL во время выполнения. Пакет даёт базовые репозитории, инструменты чтения списков и карточек, интерфейсы фильтрации запросов, typecast-классы и фабрики для настройки Cycle.

Обнаружение сущностей, консольные команды схемы и сборка кеша находятся в `componenta/cycle-app`. Если библиотеке нужна только работа с базой во время выполнения, она должна зависеть от `componenta/cycle`, а не от app-пакета.

## Установка

```bash
composer require componenta/cycle
```

Устанавливайте `componenta/cycle-app` только в приложение, которому нужны обнаружение сущностей, консольные команды схемы или интеграция с кешем фреймворка.

## Связанные пакеты

| Пакет | Зачем нужен здесь |
|---|---|
| `cycle/orm` и `cycle/database` | Основной ORM/DBAL слой. `componenta/cycle` добавляет удобные базовые классы и интеграцию, но не заменяет Cycle. |
| `componenta/di` | Нужен для регистрации репозиториев, фабрик и typecast-сервисов в контейнере. |
| `componenta/cqrs` | Часто использует репозитории в command handlers и `DataFetcher` в query handlers. |
| `componenta/paginator` | Используется read-side выборками, когда нужно вернуть slice данных и информацию о навигации. |
| `componenta/cycle-app` | Сканирует классы, строит ORM-конфигурацию и подключает консольные команды. |

## Репозитории

`Repository` расширяет репозитории Cycle ORM и даёт репозиториям приложения единую базу. Используйте репозитории для сохранения агрегатов и поиска по идентификатору.

Доменные переходы должны оставаться в агрегатах и обработчиках команд. Репозиторий не должен превращаться в сервис прав доступа или workflow-сервис.

## Чтение списков и карточек

`DataFetcher` помогает строить read-side выборки на Cycle DBAL: списки, карточки, таблицы админки и API-ответы.

Он умеет:

- определять имя таблицы с возможностью переопределения;
- выбирать колонки по интерфейсам запроса;
- применять пагинацию и опциональный точный total count;
- применять сортировку;
- маппить поля из `snake_case` в `camelCase`;
- преобразовывать значения через кастеры;
- вызывать методы вида `map{FieldName}`;
- батчем загружать связи belongs-to, has-many и many-to-many.

Методы fetcher превращают строки базы в массивы или projection objects. Связи загружаются батчем после сбора строк, поэтому выборки могут избегать N+1 запросов.

## Интерфейсы запросов

Read query object включает нужное поведение через маленькие интерфейсы:

| Интерфейс | Что включает |
|---|---|
| `PaginableInterface` | `limit` и `offset` пагинацию. |
| `RequiresTotalCountInterface` | Точный total count. |
| `SortableInterface` | Сортировку. |
| `SelectableInterface` | Явный набор колонок. |
| `SearchableInterface` | Поиск через фильтр, заданный fetcher-классом. |
| `DateRangeInterface` | Фильтрацию по интервалу дат. |
| `ListQueryInterface` | Общий интерфейс-маркер для объектов запроса списка. |

Fetcher применяет только то поведение, которое запросил объект запроса. Например, бесконечная прокрутка может не делать total-count запрос, если query не реализует `RequiresTotalCountInterface`.

## Точки расширения fetcher-класса

Прикладные fetcher-классы обычно наследуют `DataFetcher` и настраивают его через protected-свойства и методы:

| API | Назначение |
|---|---|
| `$excluded` | Поля, которые нужно удалить из результата после преобразования. |
| `$autoMapToCamelCase` | Автоматически преобразует ключи результата из `snake_case` в `camelCase`. |
| `$fieldMapping` | Явное соответствие поле БД -> поле результата. |
| `$casters` | Соответствие поле -> caster, которое резолвится через `CasterProviderInterface`. |
| `map{FieldName}()` | Метод маппинга для отдельного поля, вызывается после caster-преобразования. |
| `loadRelations()` | Необязательная точка расширения для батчевой загрузки belongs-to, has-many и many-to-many связей. |

Стандартные фильтры: `PaginationFilter`, `OrderFilter`, `SearchFilter`, `DateRangeFilter`, `DateTimeFilter`, `ComparisonFilter`, `BetweenFilter`, `InFilter`, `NotInFilter`, `NullFilter`, `NotNullFilter`, `BooleanFilter` и `SoftDeleteFilter`.

## Typecasts и фабрики

Пакет включает typecast-классы и фабрики для настройки Cycle ORM. Их можно связать вручную или через `Componenta\Cycle\ConfigProvider` в DI-конфигурации.

Встроенные typecast-классы: `UuidTypecast`, `EnumTypecast` и `CarbonTypecast`.

`ConfigProvider` регистрирует фабрики Cycle DBAL/ORM для конфигурации БД, менеджера БД, ORM, схемы, менеджера сущностей, миграций и стандартной core factory. Основные ключи конфигурации находятся в `ConfigKey`.

Если приложение использует автоматическое обнаружение сущностей и кеш ORM-конфигурации, эта часть подключается через `componenta/cycle-app`.

## Производительность

`DataFetcher` кеширует найденные методы маппинга на уровне экземпляра, хранит запрошенные наборы связей в `WeakMap` и использует оконный `count` в одном SQL-запросе, когда это безопасно. Если join может дублировать строки, fetcher переходит на отдельный `distinct count`, чтобы сохранить корректную пагинацию.

## Границы

`componenta/cycle` содержит поведение работы с базой во время выполнения. Сканирование файловой системы, обнаружение атрибутов, консольные команды схемы и интеграция компиляции принадлежат `componenta/cycle-app`.
